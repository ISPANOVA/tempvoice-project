# 🎙️ TempVoice Clone — بوت ديسكورد مجاني بالكامل للقنوات الصوتية المؤقتة

شرح كامل وشامل لكل جزء في المشروع، من الصفر للنشر، بدون أي حاجة مدفوعة أو مقفولة.

> **المشروع بالكامل (بوت + داشبورد) يعمل كـ "خدمة Railway واحدة فقط"** — لا تدفع مرتين ولا تربط خدمتين منفصلتين. البوت والداشبورد يعملان في نفس عملية Node.js الواحدة، ويتشاركان نفس قاعدة بيانات PostgreSQL. كل هذا داخل **ريبو GitHub واحد**.

## 🧠 الفكرة الأساسية — كيف تعمل "الخدمة الواحدة"؟

بدل ما يكون عندك خدمتين على Railway (واحدة للبوت، وواحدة للداشبورد)، فيه ملف واحد اسمه **`server.js`** في جذر المشروع يقوم بكل شيء بالترتيب التالي عند بدء التشغيل:

1. **يشغّل البوت أولًا**: يتصل بقاعدة البيانات، يسجّل دخوله إلى Discord، ويبقى متصلًا بـ Gateway في الخلفية (نفس سلوك أي بوت ديسكورد عادي).
2. **يسجّل مرجع البوت في الذاكرة المشتركة** (`global.__tempvoiceBotService`) — وهذا هو "الخط" الذي يصل الداشبورد من خلاله لبيانات ديسكورد الحيّة (أسماء السيرفرات، القنوات، الرتب) دون أي طلب شبكة، لأنهما الآن في نفس عملية Node.js الواحدة.
3. **يشغّل تطبيق Next.js (الداشبورد) بعد ذلك**، كـ HTTP server عادي على نفس البورت الذي توفره Railway (`process.env.PORT`).

النتيجة: طلب واحد لمتصفحك على نفس الدومين يخدمه نفس السيرفر الذي يشغّل البوت في الخلفية — **عملية واحدة، خدمة واحدة، فاتورة واحدة.**

### كيف يتم إنشاء القنوات الصوتية المؤقتة فعليًا (نفس المنطق دائمًا)
1. الأدمن يكتب `/setup` مرة واحدة في السيرفر (الأمر الوحيد الموجود في البوت بالكامل).
2. البوت تلقائيًا يعمل **Category**، قناة نصية فيها واجهة تحكم بـ 15 زرار، وقناة صوتية **Creator Channel**.
3. أي عضو يدخل الـ Creator Channel، يحصل فورًا على قناة صوتية خاصة جديدة باسمه، يتحكم فيها بالكامل من الأزرار (الاسم، الحد الأقصى، الخصوصية، الثقة/الحظر، نقل الملكية، حذفها...).
4. لما القناة تفضى، يحذفها البوت تلقائيًا ويحفظ آخر إعدادات صاحبها لتُطبَّق تلقائيًا في قناته القادمة (لو الميزة مفعّلة).
5. **الداشبورد** هو واجهة ويب لتخصيص كل هذه السلوكيات دون الحاجة لأي أوامر إضافية في ديسكورد.

## 🗄️ قاعدة البيانات — شرح كامل من الصفر

المشروع يستخدم **PostgreSQL** — قاعدة بيانات واحدة على Railway، يقرأ منها ويكتب فيها البوت والداشبورد معًا (وهما أصلًا في نفس العملية، فهذا أمر طبيعي تمامًا).

### لماذا PostgreSQL بالتحديد؟
- متوفرة مجانًا كـ "Plugin" جاهز في Railway بضغطة واحدة.
- تدعم نوع البيانات `TEXT[]` (مصفوفات) المستخدم لتخزين قوائم مثل "الأعضاء الموثوقين" أو "الرتب المسموحة" بدون الحاجة لجدول علاقات إضافي معقد.
- أداء وثبات عاليين، وهي القاعدة شبه المعيارية لمشاريع Node.js + Next.js.

### الجداول (موجودة بالكامل في `bot/src/database/schema.sql`)

| الجدول | الغرض |
|---|---|
| `guilds` | إعدادات كل سيرفر (اللغة، السينسور، قناة اللوج، التحديد العمري) |
| `creator_channels` | كل "Creator Channel" وإعداداتها التفصيلية (الاسم الافتراضي، الكاتيجوري، الليميت، البرايفسي، الرولز المسموحة...) |
| `temp_channels` | كل قناة مؤقتة شغالة دلوقتي فعليًا في ديسكورد (آيدي القناة، المالك، حالة الخصوصية) |
| `user_channel_settings` | الإعدادات المحفوظة لكل عضو (تُستخدم في "استرجاع الإعدادات") |
| `channel_relations` | جدول مساعد للعلاقات بين القنوات والأعضاء (ثقة/حظر) |
| `dashboard_users` | بيانات مستخدمي الداشبورد القادمة من Discord OAuth2 (التوكنات...) |
| `action_logs` | سجل عمليات اختياري للمستقبل (تتبع كل حدث) |

### كيف يتم إنشاء الجداول؟
**لا حاجة لأي أمر يدوي.** عند أول تشغيل، الدالة `initDatabase()` (تُستدعى من `bot/src/index.js` فور بدء `server.js`) تُنفّذ ملف `schema.sql` بالكامل تلقائيًا باستخدام `CREATE TABLE IF NOT EXISTS`، وهي عملية آمنة تمامًا حتى لو تكررت.

### كيف يصل الداشبورد لبيانات Discord الحيّة (اسم السيرفر، قنواته، رتبه)؟
**لا توجد قاعدة بيانات لهذه البيانات لأنها تتغير باستمرار.** بدلاً من ذلك:
- البوت (المتصل بـ Discord Gateway) يحتفظ بهذه البيانات في الذاكرة، ويسجّل دوال الوصول إليها على `global.__tempvoiceBotService` (انظر `bot/src/botService.js`).
- الداشبورد (`dashboard/lib/botApi.ts`) يستدعي هذه الدوال **مباشرة من الذاكرة المشتركة** دون أي طلب HTTP، لأنهما في نفس عملية Node.js تمامًا.

## 🚀 خطوات الإعداد الكاملة — ريبو واحد، خدمة Railway واحدة

### الخطوة 1: إنشاء تطبيق ديسكورد (Discord Application)

1. روح على [Discord Developer Portal](https://discord.com/developers/applications)
2. اضغط **New Application** واختار اسم (مثلاً `TempVoice`)
3. من القائمة الجانبية اضغط **Bot** → اضغط **Reset Token** وانسخ الـ Token (هتحتاجه كـ `DISCORD_TOKEN`)
4. في نفس صفحة Bot، فعّل هذه الصلاحيات تحت **Privileged Gateway Intents**:
   - ✅ `SERVER MEMBERS INTENT`
   - ✅ `MESSAGE CONTENT INTENT`
5. من القائمة الجانبية اضغط **OAuth2** → **General** وانسخ:
   - `Client ID` (هتحتاجه كـ `CLIENT_ID` و `DISCORD_CLIENT_ID` — نفس القيمة بالضبط في المكانين)
   - اضغط **Reset Secret** وانسخ `Client Secret` (هتحتاجه كـ `DISCORD_CLIENT_SECRET`)
6. لسه في صفحة OAuth2، تحت **Redirects** اضغط **Add Redirect** وضيف مؤقتًا (سنعدلها بالرابط الصحيح بعد النشر):
   ```
   https://placeholder.up.railway.app/api/auth/callback
   ```

### الخطوة 2: بناء رابط دعوة البوت للسيرفر

من نفس صفحة **OAuth2 → URL Generator**:
- في **Scopes** اختار: `bot` و `applications.commands`
- في **Bot Permissions** اختار:
  - `Manage Channels`, `Manage Roles`, `Move Members`, `Mute Members`, `Deafen Members`
  - `View Channels`, `Connect`, `Speak`, `Send Messages`, `Create Instant Invite`, `Manage Webhooks`
- خد الرابط الناتج في الأسفل واحفظه — هتحتاجه كـ `NEXT_PUBLIC_BOT_INVITE_URL`

### الخطوة 3: رفع المشروع كاملاً (بوت + داشبورد + server.js) في ريبو واحد على GitHub

من داخل مجلد المشروع الرئيسي `tempvoice-project` (الذي يحتوي على `server.js`، `bot/`، و `dashboard/` بجانب بعض):

```bash
cd tempvoice-project
git init
git add .
git commit -m "Initial commit - TempVoice unified bot + dashboard"
git branch -M main
git remote add origin https://github.com/YOUR_USERNAME/tempvoice-project.git
git push -u origin main
```

> ✅ تأكد أنك تـ push للمجلد **الجذر** بالكامل (وليس مجلد `bot` أو `dashboard` فقط)، لأن `server.js` و `package.json` الموحّد يقعان في الجذر، وهما ما يجعلان الخدمة الواحدة تعمل.

### الخطوة 4: إنشاء مشروع Railway وقاعدة البيانات

1. افتح [Railway](https://railway.app) وسجّل حساب (مجاني، فيه باقة تجريبية شهرية)
2. اضغط **New Project** → **Provision PostgreSQL**
3. هذه الخدمة ستبقى بجانب خدمة التطبيق التي ستنشئها الآن — كل مشروع Railway يحتوي خدمة Postgres وخدمة التطبيق، وهذا لا يُحسب كـ"خدمتين تدفع لهما بشكل منفصل" بالمعنى الذي تقلق منه؛ Railway يحسب حسب الاستخدام الفعلي (موارد) لا حسب عدد الخدمات، وقاعدة البيانata ضرورية لتشغيل أي بوت كهذا

### الخطوة 5: سحب الريبو لإنشاء خدمة التطبيق الواحدة

1. من نفس مشروع Railway، اضغط **New** → **GitHub Repo**
2. لو أول مرة تربط GitHub بـ Railway، هيطلب منك صلاحية الوصول — وافق واختار الريبو `tempvoice-project`
3. Railway سينشئ **خدمة واحدة فقط** لهذا الريبو. **لا تحدد Root Directory هذه المرة** — اتركه فاضي/على الجذر، لأن `package.json` و `server.js` الموحّدين موجودان أصلًا في جذر الريبو
4. روح على تاب **Settings** للخدمة، وتأكد أن:
   - **Build Command**: `npm install && npm run build` (موجود تلقائيًا من `railway.json`)
   - **Start Command**: `npm start` (يشغّل `node server.js`)
5. روح لتاب **Variables** وضيف كل المتغيرات من `.env.example`:
   - `DISCORD_TOKEN`, `CLIENT_ID` (من الخطوة 1)
   - `DATABASE_URL` → اضغط **Add Reference** واختار خدمة Postgres (لا تكتبها يدويًا، Railway يربطها تلقائيًا)
   - `SESSION_SECRET` → نص عشوائي طويل (يمكن توليده محليًا بالأمر `openssl rand -base64 32`)
   - `DISCORD_CLIENT_ID`, `DISCORD_CLIENT_SECRET` (من الخطوة 1)
   - `NEXT_PUBLIC_BOT_INVITE_URL` (من الخطوة 2)
   - `DISCORD_REDIRECT_URI` → سنضيفها بالقيمة الصحيحة في الخطوة القادمة
6. روح لتاب **Settings → Networking** واضغط **Generate Domain** — هذا هو الدومين العام لكل من البوت والداشبورد معًا (مثلاً `https://tempvoice-production-xxxx.up.railway.app`)
7. رجع لتاب **Variables** وأضف:
   ```
   DISCORD_REDIRECT_URI=https://tempvoice-production-xxxx.up.railway.app/api/auth/callback
   ```
   (استخدم الدومين الحقيقي بتاعك بدل المثال)
8. تابع تاب **Deployments → Logs** وانتظر ظهور:
   ```
   ✅ البوت متصل وجاهز.
   ✅ الداشبورد جاهز.
   🚀 الخدمة الموحّدة (بوت + داشبورد) تعمل على المنفذ ...
   ```

### الخطوة 6: تحديث رابط الـ Redirect في Discord Developer Portal

1. رجع لـ [Discord Developer Portal](https://discord.com/developers/applications) → تطبيقك → **OAuth2**
2. تحت **Redirects**، احذف الرابط المؤقت من الخطوة 1 وضيف الرابط الحقيقي من الخطوة 5.7
3. احفظ التغييرات

### الخطوة 7: تسجيل أمر `/setup` على ديسكورد

الأمر يُسجَّل مرة واحدة فقط (وليس عند كل تشغيل). من جهازك (محتاج Node.js مثبت محليًا، ونفس قيمتي `DISCORD_TOKEN` و `CLIENT_ID`):

```bash
cd tempvoice-project
npm install
cp .env.example .env
# افتح .env واملأ DISCORD_TOKEN و CLIENT_ID فقط (الباقي غير مهم لهذا الأمر)
npm run deploy-commands
```

سيظهر أمر `/setup` في كل سيرفرات ديسكورد التي يدخلها البوت خلال دقائق.

### الخطوة 8: دعوة البوت لسيرفرك وتجربة كل شيء

1. استخدم الرابط من الخطوة 2 لدعوة البوت لسيرفر ديسكورد بتاعك
2. اكتب `/setup` في السيرفر
3. افتح رابط الداشبورد (نفس الدومين من الخطوة 5.6) وسجّل دخولك بحساب ديسكورد، واختار السيرفر، وجرّب التخصيص

### ✅ خلاصة الفكرة في جملة واحدة

ريبو واحد على GitHub فيه `server.js` في الجذر يجمع البوت والداشبورد، وعلى Railway بتعمل **خدمة تطبيق واحدة فقط** (بجانب خدمة Postgres الضرورية لتشغيل أي قاعدة بيانات) — وأي `git push` جديد يُحدّث الخدمة كلها تلقائيًا دفعة واحدة.

## 🛠️ شرح القرارات التقنية المهمة

### لماذا خدمة Railway واحدة بدل خدمتين؟
بناءً على طلبك، البوت والداشبورد يعملان كآن في **نفس عملية Node.js** عبر ملف `server.js` في الجذر. هذا يعني فاتورة استضافة واحدة بدل فاتورتين، مع الحفاظ على كل الوظائف بدون أي فقدان في الأداء — الاتصال بين الطرفين أصبح أسرع فعليًا (استدعاء دالة مباشر في الذاكرة بدل طلب HTTP عبر الشبكة).

### كيف يتواصل الداشبورد مع بيانات Discord بدون API؟
ملف `bot/src/botRegistry.js` يحتفظ بمرجع لـ Discord Client بعد جاهزيته. `server.js` يأخذ دوال الوصول من `bot/src/botService.js` ويسجّلها على `global.__tempvoiceBotService`. الداشبورد (`dashboard/lib/botApi.ts`) يقرأ من هذا الـ global مباشرة. لا توجد طبقة شبكة بينهما، ولا حاجة لمفتاح سري (`INTERNAL_API_SECRET`) كان موجودًا في النسخة السابقة بخدمتين منفصلتين.

### لماذا `global` بدل `require` مباشر لملفات بعضهما؟
جرّبنا فعليًا استخدام `require('../../bot/src/botService.js')` مباشرة من كود الداشبورد، لكن Next.js (خصوصًا مع Turbopack) يتتبع ويجمّع (bundle) كل استدعاءات `require` ضمن حدود مجلد `dashboard/` فقط، فالوصول لملف خارج هذا المجلد لم يكن موثوقًا في كل أوضاع التشغيل. الحل العملي المضمون: تسجيل الدوال على `global` وقت التشغيل الفعلي من `server.js` (الذي لا يمر بأي bundler أصلًا، فهو ملف Node.js عادي).

### لماذا npm workspaces؟
لتجميع كل الحزم (discord.js، next، pg، إلخ) في `node_modules` واحد فقط على مستوى الجذر، بدل تكرارها في `bot/node_modules` و `dashboard/node_modules` بشكل منفصل. هذا يقلل حجم المشروع المرفوع، ويضمن أن كل من `server.js` والكود الداخلي يستخدمان نفس نسخ الحزم بالضبط.

### كيف تُمنع التعديلات غير المصرّح بها في الداشبورد؟
كل API Route (`dashboard/app/api/guilds/...`) يستخدم `requireGuildAccess()` الذي:
1. يتحقق من وجود جلسة صالحة
2. يسحب قائمة سيرفرات المستخدم **مباشرة من Discord API** (وليس من قاعدة بيانات قد تكون قديمة)
3. يتحقق أن المستخدم يملك صلاحية `MANAGE_GUILD` أو `ADMINISTRATOR` على هذا السيرفر بالتحديد

## ❓ استكشاف الأخطاء الشائعة

| المشكلة | الحل |
|---|---|
| ظهور خطأ "خدمة البوت غير متاحة بعد" في الداشبورد | تأكد أنك تُشغّل المشروع عبر `npm start` (الذي يستدعي `server.js`) من الجذر، وليس `next start` بمفردها داخل `dashboard/` |
| `/setup` لا يظهر في ديسكورد | تأكد من تشغيل `npm run deploy-commands` مرة واحدة على الأقل، وانتظر دقائق قليلة |
| الداشبورد يعطي خطأ "لا تملك صلاحية إدارة هذا السيرفر" | تأكد أن حسابك فيه صلاحية Manage Server الفعلية على السيرفر في ديسكورد نفسه |
| الداشبورد لا يعرض أي سيرفرات | تأكد أن البوت منضم فعليًا لنفس السيرفر، وأن خدمة Railway نفسها (البوت) متصلة بنجاح بحسب الـ Logs |
| القنوات الصوتية لا تُنشأ عند دخول Creator Channel | تأكد من تفعيل `SERVER MEMBERS INTENT` في Discord Developer Portal، وأن البوت لديه صلاحية Manage Channels في السيرفر |
| رسالة "Database connection failed" | تأكد من نسخ `DATABASE_URL` بالكامل بدون مسافات إضافية (الأفضل استخدام Add Reference بدل النسخ اليدوي)، ومن أن قاعدة بيانات Railway تعمل |
| خطأ بناء على Railway متعلق بـ Turbopack/workspace root | تأكد أنك رفعت الجذر بالكامل (وليس مجلد فرعي فقط)، وأن `dashboard/next.config.ts` موجود وفيه `turbopack.root` |
| الصفحة تفضل على "Loading..." والسيرفرات لا تظهر، وفي الـ Logs ظهرت رسائل عن `webpack-hmr` أو `allowedDevOrigins` | هذا يعني أن التطبيق يعمل في وضع تطوير (`next dev`) بدل وضع إنتاج (`next start`) خطأً. تأكد من إضافة `NODE_ENV=production` بشكل صريح في Variables على Railway، وتأكد أن `npm run build` نُفِّذ بنجاح قبل التشغيل |
